这篇文章整理自知识库里的 `passwall.md`、`passwall详细设置.md`、`passwall快速运维.md` 和问题记录。原始资料里有不少重复内容，我把它合并成一份更适合日常复用的 PassWall 运维手册。

目标很明确：在 OpenWrt / ImmortalWrt 软路由上，通过 PassWall 做尽可能完整的网段代理，减少 DNS、IPv6、UDP、WebRTC 等常见泄露点。

> 注意：本文只保留公开安全的配置方法和排障思路，不包含真实节点、订阅、账号密码或业务线路信息。

## 为什么放在软路由上做

在 PC 或手机上开代理客户端时，即使选择了“全局模式”，仍然可能出现这些绕过点：

- 系统后台流量没有进入代理。
- UDP 流量没有被完整接管。
- DNS 查询走了本地运营商。
- IPv6 直连绕过了 IPv4 代理。
- WebRTC 暴露内网地址或真实公网地址。

软路由的优势是可以从网关层接管指定设备或网段的流量。终端设备不需要理解代理，只要把 OpenWrt 当网关，流量就会被统一处理。

## 推荐配置模板

下面是一套偏“零泄露”的配置模板，适合对流量一致性要求比较高的场景。

| 配置项 | 推荐值 | 作用 |
| --- | --- | --- |
| 访问控制状态 | 启用 | 让规则真正生效 |
| 源地址 | 具体设备 IP 或网段 | 控制哪些设备走代理 |
| 切换模式 | 全局代理 | 减少分流判断导致的漏流 |
| 中国列表 | 关闭 | 避免本地直连兜底 |
| TCP 转发端口 | `1:65535` | 接管所有 TCP 端口 |
| UDP 转发端口 | `1:65535` | 接管语音、游戏、QUIC 等 UDP 流量 |
| UDP 转发屏蔽端口 | 不使用 | 全局模式下不要额外屏蔽 |
| DNS 分流 | `Dnsmasq` 或 `UDP 节点解析` | 配合过滤引擎处理 DNS |
| 过滤模式 | `Xray` 或 `Sing-Box` | 让代理核心处理 DNS |
| DNS 协议 | `TCP` | 降低 UDP DNS 泄露和劫持 |
| 远程 DNS | 按出口地区选择 | 海外出口可用 `8.8.8.8` / `1.1.1.1` |
| EDNS Subnet | 留空 | 避免主动暴露地理位置 |
| 过滤代理域名 IPv6 | 开启 | 防止 IPv6 直连 |

## 源地址要固定

访问控制的源地址可以写单个设备，也可以写网段：

```text
10.10.200.50
10.10.200.0/24
```

如果只想让某几台设备走代理，建议先在 OpenWrt 的静态地址分配里绑定 MAC 和 IP。否则设备 IP 一变，访问控制规则就可能失效。

## 全局代理和分流模式怎么选

如果目标是“尽量不泄露”，优先选择全局代理。

分流模式本身需要做域名/IP 判断，这个判断过程很容易牵涉 DNS 预解析、规则命中失败、国外 IP 直连等边缘情况。全局代理虽然牺牲了一些精细度，但排障更直接。

常见取舍：

- 全局代理：适合零泄露、固定业务设备、海外回国或统一出口。
- GFW 列表：适合普通上网，不适合强约束场景。
- 中国列表以外：体验折中，但仍然依赖规则和 DNS 质量。

## DNS 配置思路

DNS 是最容易漏的地方。

海外出口常用：

```text
8.8.8.8
1.1.1.1
```

回国出口常用：

```text
223.5.5.5
119.29.29.29
```

海外回国时，远程 DNS 建议改成国内大厂 DNS。原因是国内视频网站、应用商店、CDN 服务会根据 DNS 地理位置分配资源。如果 DNS 仍然是海外，可能出现版权区域错误、CDN 分配不佳或访问慢。

## IPv6 和 WebRTC

如果出口链路只保证 IPv4，建议同时处理 IPv6：

- PassWall 中开启过滤代理域名 IPv6。
- 终端侧临时关闭 IPv6 做对照测试。
- 验证浏览器 WebRTC 是否暴露真实地址。

WebRTC 检查时重点看 SDP 日志里的三类字段：

```text
c=           连接地址
a=candidate ICE 候选地址
m=           媒体流类型和传输协议
```

如果 `c=` 行出现真实公网 IP，需要重点排查。  
如果 `a=candidate` 里出现内网 IP，虽然通常无法直接定位，但仍然可能成为浏览器指纹的一部分。  
如果 `m=` 里看到 UDP，也说明 WebRTC 对 UDP 的依赖很强，不应该随意屏蔽 UDP。

## 备份和恢复

在 Mac 上重新打包 PassWall 备份时，注意不要把 macOS 的 `._` 隐藏属性文件打进去。

推荐打包方式：

```bash
cd /path/to/passwall-backup
COPYFILE_DISABLE=1 tar -czvf new_passwall_backup.tar.gz etc usr
```

这里的关键点：

- `COPYFILE_DISABLE=1`：禁止 macOS 生成 `._` 元数据文件。
- `etc usr`：只打包需要的目录，不额外套一层外部目录。

恢复后建议执行：

```bash
uci commit passwall
/etc/init.d/passwall restart
```

如果恢复后配置没有生效，优先检查：

- 压缩包里是否多套了一层目录。
- 是否包含 `/etc/config/passwall`。
- 恢复后是否提交并重启服务。
- 备份里是否混入无关文件。

## 常见故障排查

### 配置后彻底断网

优先检查：

1. 切换模式是否选成了 `GFW 列表`。
2. 节点是否在线。
3. DNS 设置是否和过滤模式冲突。
4. IPv6 是否绕过。
5. 访问控制源地址是否写错。

如果使用 `Xray TCP DNS`，但切换模式还是 `GFW 列表`，可能出现逻辑死锁。先改回全局代理做排障。

### IP 变了，但 DNS 测试仍然显示本地服务器

检查：

- PassWall DNS 设置里是否勾选了“使用系统 DNS”。
- 访问控制里的源地址是否包含当前设备。
- 终端是否手动配置了其他 DNS。
- 浏览器是否启用了自己的安全 DNS。

### 网速很慢

排查顺序：

1. 在 OpenWrt 上用 `top` 看 CPU 是否被代理核心打满。
2. 换更轻的协议或更好的线路。
3. 测试软路由到代理入口的 ping / mtr。
4. 测试出口 IP 查询和常用网站首包时间。

CPU 飙升时，优先考虑软路由算力不足或协议加密开销过高。

## 验收三件套

配置完成后建议做这三类测试：

| 项目 | 工具 | 合格标准 |
| --- | --- | --- |
| IP 归属 | `ip.p3terx.com` / `cip.cc` | 显示目标出口 |
| DNS 泄露 | `dnsleaktest.com` Extended Test | 不出现本地运营商 DNS |
| WebRTC | `browserleaks.com/webrtc` | 不暴露真实公网 IP |

如果这三项都符合预期，基本可以认为软路由侧代理链路已经稳定进入可用状态。

## 日常维护清单

- 新设备接入后，先确认静态 IP 绑定。
- 修改访问控制后，检查源地址是否命中。
- 修改 DNS 后，重启 PassWall 并重新做 DNS 泄露测试。
- 每次恢复备份后执行 `uci commit passwall` 和服务重启。
- 线路不稳定时先区分“本地网络问题”和“代理入口问题”。
- 对重要设备定期跑节点巡检脚本，保留 CSV/JSONL 结果。
